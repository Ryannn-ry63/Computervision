# 复现图片索引与环境说明

## 先说明

图片文件本身没有被重命名。报告中使用“图表 5、图表 6……”是为了正文引用更规范，实际图片仍保存在 `outputs/` 和 `multisensor_fusion_reproduction/outputs/` 中。

我在上传 GitHub 前改名的是三个子项目内部的 Git 元数据目录：

```text
CLOCs/.git -> CLOCs/.git.local-backup
PointPainting/.git -> PointPainting/.git.local-backup
mmdetection3d/.git -> mmdetection3d/.git.local-backup
```

这样做只是为了让 `computervision` 作为一个总仓库上传时，三个子项目的源码能作为普通文件进入 GitHub，而不是被 Git 识别成 submodule。它不影响图片文件。`.git.local-backup` 已加入 `.gitignore`，不会上传到 GitHub。

```text

outputs/fusion_reproduction/bevfusion_result_sample10.png

```

如果只能放一张综合图，建议使用：

```text

```

它更适合快速说明相机图像、LiDAR 投影和 BEV 表示之间的关系，但它不能替代 PointPainting、BEVFusion、CLOCs 三类方法各自的机制图。

## 依赖环境差异是否仍然存在

仍然存在。

这不是你的操作问题，而是这些代码仓库本身依赖的时间跨度比较大，和当前服务器环境不完全一致：

- PointPainting 原始代码依赖旧版 OpenPCDet、MMSegmentation、MMCV 等组件。本文在独立 `pointpainting_legacy` 环境中跑通了仓库内置 MMSegmentation DeepLabV3+ painting 分支，但 detector 部分仍是单帧链路验证，不是官方完整评测。
- BEVFusion 使用当前 `motiondetection` 环境和兼容后的官方预训练权重进行可视化推理，未重新训练完整模型。
- CLOCs 原始代码依赖 SECOND-1.5 和 `spconv v1`。当前服务器无法直接回到完全一致的 `spconv v1` 栈，因此在独立 `clocs_legacy` 环境中使用 Python 3.7、PyTorch 1.7.1 和 `spconv-cu118` 的兼容路径跑通官方单帧流程。

所以报告里建议继续使用下面这个表述：

```text
受限于完整数据集组织、官方检测器权重，以及原始代码依赖栈与当前服务器环境存在版本差异，本文复现以单帧样例、官方预训练权重推理和局部链路验证为主。以下结果用于解释融合流程与工程约束，不作为 KITTI 或 nuScenes 完整评测结果。
```

不要写成：

```text
完全复现了原论文环境和 benchmark 指标。
```

更准确的说法是：

```text
本文完成了核心流程复现和单帧可视化验证，用于说明数据层、特征层和决策层融合机制。
```
